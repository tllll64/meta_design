# 前端讨论稿 v0.1

> 目的：把 RQ2 关于“可制定规则”的讨论，映射成中期 demo 可展示、可继续推进的前端界面骨架。
>
> 当前定位：这不是最终视觉定稿，而是一份用于讨论页面结构、区域关系、交互逻辑与中期实现范围的前端讨论稿。

## 1. 研究内容到前端区域的映射

RQ2 当前已经把系统内容收束为：

- `内容规则`
- `结构规则`
- `视觉规则`
- `约束`
- `分工`
- `人-AI 临时对话层`

这些内容不应在前端上以同一种形式出现。

### 1.1 内容规则

内容规则关注对象的：

- 类型
- 语义
- 角色
- 优先级
- 事实边界

更适合的前端表达：

- 对象卡片
- 字段表单
- 类型/语义/角色标签
- 内容清单和分组

### 1.2 结构规则

结构规则关注：

- 模块划分
- 层级关系
- 区域分布
- 对象关系
- 阅读路径

更适合的前端表达：

- 中央画布
- 模块分区图
- 层级树
- 连线关系图

结论：结构规则不应被做成纯表单，它应该成为中间主视区的重要内容。

### 1.3 视觉规则

视觉规则关注：

- 字体
- 色彩
- 强调方式
- 图形风格
- 容器与密度

更适合的前端表达：

- 右侧样式面板
- token 列表
- 预设选择
- 局部覆写

### 1.4 约束与分工

`约束` 与 `分工` 更像横向控制维度，不适合做成与三类规则并列的大主区。

更适合的前端表达：

- 顶部状态条
- 权限标签
- 审批/确认标记
- 局部抽屉面板

### 1.5 人-AI 临时对话层

对话层不是孤立的聊天框，而是任务中的即时共创区。

它至少应支持：

- 草案生成
- 规则建议
- 结构探索
- 局部修补
- 保存为规则

更适合的前端表达：

- 可折叠侧栏
- 候选结果区
- “加入任务 0”/“保存为规则”按钮

## 2. 中期 demo 的前端目标

中期 demo 不需要把整套系统全部做完，但需要把“机制”讲明白。

### 2.1 中期主界面应完成的目标

中期 demo 的主界面需要在短时间内让评审理解：

1. 这是一个围绕规则机制展开的研究原型
2. 它如何区分元设计层、`任务 0` 与正式任务
3. 它为什么需要内容/结构/视觉三类规则
4. 人和 AI 在哪里协作，系统在哪里做诊断
5. 用户如何在同一工作台里完成预演、回改与继续执行

### 2.2 中期阶段不强求的内容

当前可不优先实现：

- 完整后端联动
- 真实多用户权限系统
- 复杂资产管理
- 高保真生成能力
- 所有规则项的完整编辑

中期 demo 更重要的是：

- 结构说得通
- 页面关系说得通
- 用户路径说得通
- 原型状态之间切换说得通

## 3. 建议的界面层级

### 3.1 中期阶段：直接进入工作台

中期 demo 不需要单独做一个“首页 / 系统概览页”。

更合适的方式是：打开原型后直接进入主工作台，在工作台内部完成系统机制的展示与操作。

这样更符合中期阶段的目标：

- 不把精力消耗在介绍页
- 直接展示研究原型如何工作
- 让评审尽快看到规则、任务、对话、诊断之间的关系

### 3.2 后期阶段：视测试需求补一个入口层

如果后续进入测试阶段，尤其是需要区分不同测试项目、不同任务样例或不同受试者路径时，可以再补一个更轻量的入口层。

这个入口层更像：

- 测试项目选择器
- 原型任务列表
- 不同 case 的启动入口

它的职责不是介绍研究背景，而是帮助测试人员进入正确的测试项目。

### 3.3 工作台的职责

工作台用于承载真正的交互与操作逻辑，也是中期 demo 的主界面。

它的职责是：

- 配置规则
- 查看结构
- 执行任务 0
- 与 AI 对话协作
- 预览结果
- 诊断并回改

## 4. 工作台布局草案

建议采用 `左 - 中 - 右 + 次级侧栏` 的可折叠工作台结构。

### 4.1 左侧：阶段与导航

作用：回答“我现在在哪个阶段”

建议包含：

- 系统名称 / 当前任务名
- `元设计层`
- `任务 0`
- `正式任务`
- 规则库
- 模板库

这一栏适合做成可收缩导航栏。

### 4.2 中央：主工作区

作用：回答“当前正在处理什么”

这是整个界面的主视觉，不建议折叠。

在不同阶段显示不同内容：

- `元设计层`：规则结构总览、模板骨架、系统关系图
- `任务 0`：试跑结果、预演产物、规则生效情况
- `正式任务`：最终任务结果、模块化产物、当前版本内容

中央区应承载：

- 结构规则的主要可视化
- 当前任务结果
- 关键预览

### 4.3 右侧：属性与规则面板

作用：回答“当前对象可以怎么被修改”

建议按当前选中对象切换内容：

- 选中内容对象时：显示类型/语义/角色/优先级
- 选中结构模块时：显示模块关系、布局、层级
- 选中视觉对象时：显示字体、色彩、强调、密度

这一栏适合做成：

- 默认展开
- 可折叠
- 支持按组展开/收起

### 4.4 次级侧栏：对话与诊断

作用：回答“AI 现在能帮什么”“系统发现了什么问题”

建议作为第二层可开合面板存在，而不是永久抢占布局。

可承载：

- 人-AI 对话
- 候选方案
- 规则建议
- 生效诊断
- 缺失提醒
- 保存为规则

这一栏在不同阶段的优先级不同：

- `元设计层`：规则建议、结构探索优先
- `任务 0`：诊断、生效检查、快速回改优先
- `正式任务`：草案生成、局部修补、候选方案优先

## 5. `任务 0` 的界面定义

`任务 0` 是这次讨论里的关键界面概念。

它不是一个普通的正式任务，也不是后台测试页，而是：

`正式任务开始前，用于验证机制的一次预演任务`

### 5.1 任务 0 的目标

- 用最小输入快速触发一次生成
- 检查规则有没有生效
- 发现规则缺失、冲突、过度模糊的位置
- 允许直接回写到元设计规则

### 5.2 任务 0 的最小流程

1. 用户选择一个轻量任务模板
2. 输入少量任务条件
3. 系统基于当前规则试跑
4. 用户查看预演结果与诊断提示
5. 用户决定：
   - 回改规则
   - 保存临时规则
   - 进入正式任务

### 5.3 任务 0 在界面上的表现

建议包含三个并列信息块：

- `任务输入`
- `预演结果`
- `规则诊断`

这三块不一定同时大展开，但必须都能被快速访问。

## 6. 可折叠区域的原则

折叠不是为了隐藏功能，而是为了减轻认知负担。

### 6.1 适合折叠的区域

- 规则库
- 模板库
- 对话区
- 诊断区
- 候选结果区
- 约束与权限细项

### 6.2 不适合折叠掉的区域

- 当前阶段标识
- 中央主工作区
- 当前选中对象的核心上下文
- 最关键的主操作入口

### 6.3 折叠逻辑建议

- 默认只展开当前任务最相关的 1 到 2 个工具区
- 其他区域保持 peek 状态或图标态
- 区域展开后应明确回答一个问题，而不是只展示一堆控件

可用的设计语言：

- 折叠栏
- 抽屉
- 标签切换
- 可收缩面板组
- 局部 pin / unpin

## 7. 中期 demo 推荐的页面组织

如果从“展示路径”角度考虑，中期 demo 可以先只做一个主工作台页面。

### 7.1 页面一：任务工作台

建议优先演示：

- 左侧阶段导航
- 中央视图
- 右侧规则面板
- 可展开的对话/诊断区
- `任务 0` 的预演逻辑

这样中期 demo 就已经能讲清：

- 研究问题如何被映射到界面
- 前端工作台的基本框架
- 用户如何在同一界面里切换阶段
- 机制如何被试跑、诊断和回改

### 7.2 后续可补：测试项目入口层

如果后续需要面向测试人员组织多个样例任务，可以再追加一个轻量入口页。

它更适合承担：

- 区分不同测试项目
- 区分不同任务样例
- 为测试人员提供明确入口

这属于后续测试组织层，不属于中期 demo 的必需范围。

## 8. 当前推荐的中期实现范围

如果要尽快把前端跑起来，建议先实现“结构对”的版本。

### 8.1 必做

- 系统概览首页
- 工作台基础布局
- 左中右三栏关系
- 至少一个可折叠辅助面板
- `任务 0` 的静态流程展示
- 三类规则的占位内容

### 8.2 可用静态假数据完成

- 对象卡片
- 模块结构图
- 视觉规则列表
- 诊断提示
- 对话建议
- 预演结果卡片

### 8.3 可以后补

- 真实 AI 生成
- 真正可编辑的复杂画布
- 完整规则持久化
- 后端联动
- 多任务管理

## 9. 仍需继续讨论的问题

下面这些问题建议作为后续讨论重点：

1. 首页更偏“研究原型说明页”还是“直接进入工作台的启动页”
2. 工作台中央区在 `元设计层` 时更像流程图，还是更像结构画布
3. `任务 0` 是否需要独立模板列表，还是直接复用正式任务入口
4. 对话区和诊断区是合并为一个侧栏，还是拆成两个可切换 tab
5. 中期 demo 中，结构规则的可视化到底做到什么粒度最合适
6. 首页与工作台之间是否需要一个过渡页面

## 10. 一个可执行的收束方向

如果现在要继续往下推进，我建议我们下一轮优先敲定这三件事：

1. 首页的版块顺序
2. 工作台的左中右区域具体承载什么
3. `任务 0` 在工作台里到底如何进入、如何退出

这三点一旦定下来，后面的视觉风格、组件形式和静态原型实现都会容易很多。
