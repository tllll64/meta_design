# RQ2

## 研究问题

**RQ2：在信息设计场景下，具体任务之外，设计师可以制定怎样的元设计空间？**

<br />

## 核心结论

`RQ2` 聚焦那些 `高影响、高歧义、高复用` 的中高层元设计空间内容，而不是穷尽所有执行细节。

<br />

## 规则分层

元设计空间按抽象层级分为三层。RQ2 的重点在 `高层` 与 `中层`；底层规则更适合以外部化载体（模板、预设方案、skill、检查清单）的形式被调用，而不是作为主节点前置定义。这些规则并非先验存在，而是设计师在真实任务中不断积累、被误解倒逼、反复验证后逐渐稳定下来的——这也是高层与中层规则比底层细节更值得优先沉淀的原因：它们更抽象、更稳定，跨任务的复用价值更高。

判断一条规则是否值得前置显性化，有两个维度：一是它能不能被描述清楚，二是它足不足够稳定。P7 提出了一个精确的判断标准——`规则的半衰期`：

> 如果它半衰期越短，显性化成本越高，那是不是说明它可能就不适合让 AI 来执行，而适合人在 loop 里面去记录一下？

也就是说，即便某条规则能被语言描述，如果它随着业务迭代频繁变化，显性化的维护成本也会超过复用收益，反而不如由人在协作过程中直接把关。

P8 从工具迭代速度的角度提出了另一个判断维度：当 AI 工具本身处于快速迭代期时，今天写下的规则可能很快因工具升级而失效——"再过一段时间可能会有新的概念，我不着急"。这意味着规则显性化的投入时机，也需要参照工具稳定性来评估。

| 规则层级 | 核心作用 | 对应 RQ2 节点 |
| --- | --- | --- |
| `高层规则` | 定义任务前提、目标方向与总体判断标准，回答"这次设计为什么做、为谁做、什么算成立" | 任务目标与场景 · 设计原则 · 人机分工与控制边界 |
| `中层规则` | 定义信息如何被组织、模块如何被编排、风格如何被锚定，回答"这次设计应如何被组织与展开" | 信息内容与层级 · 结构与模块关系 · 视觉风格与参考锚点 · 跨层映射规则 |
| `底层规则` | 定义局部执行细节与重复性实现方式，回答"哪些元素怎么画、怎么排、怎么落出来" | 不作为 RQ2 主节点，寄存在 Assets / 引用层、模板、preset、layout recipe 中 |

## 元设计空间主表

| 规则层级 | 元设计空间维度 | 核心内涵 | 可结构化内容 | 典型形成方式 | 原型转译方向 |
| --- | --- | --- | --- | --- | --- |
| `高层规则` | 任务目标与场景 | 界定任务前提、使用语境与交付边界 | 受众、场景、渠道、目标、交付约束| 由设计师直接设定，或通过与 AI 的前期对话共同澄清 | 任务卡片、场景面板、任务摘要条 |
| `高层规则` | 设计原则 | 沉淀跨任务复用的高层判断标准，兼顾生成原则与评判标准 | 原则内容（生成 & 评判）、适用范围、关键映射判断 | 由设计师主动输入，并可在任务过程中持续抽象、补充与修正；AI 可辅助归纳但不替代人的原则判断 | 原则卡片、原则列表、可新增原则入口 |
| `高层规则` | 人机分工与控制边界 | 明确不同层上的主导权、介入程度与不可改动部分 | 人机参与度、锁定项 | 跟随信息、结构、视觉等层同步调节；AI 可先给出默认建议，再由设计师修正 | 分层控制栏、参与度调节、锁定标签 |
| `中层规则` | 信息内容与层级 | 明确任务中包含哪些信息，以及这些信息的主次关系 | 对象类型、对象语义、重要程度、改动权限 | 由生成结果中识别与抽取，再由设计师进行审查、修正与补充 | 信息卡片列表、层级树、内容清单 |
| `中层规则` | 结构与模块关系 | 明确内容如何被分区、排序、组织与打包 | 模块划分、模块顺序、模块主次、模块内对象组织 | 可由设计师直接组织，也可基于生成结果进行结构识别与后续修正 | 结构画布、模块编排板、结构关系图 |
| `中层规则` | 视觉风格与参考锚点 | 提供风格方向及视觉对齐依据 | 风格关键词、参考图、主视觉方向、风格元素 | 可在生成前以前置风格锚点限定，也可在初步结果基础上追加风格约束 | 风格卡片、参考图板、轻量风格摘要 |
| `高层规则` | 流程资产与经验沉淀 | 沉淀可复用的推进方式与关键检查经验 | 流程名称、适用场景、步骤列表、检查重点 | 原型阶段以人工编排为主，AI 可辅助总结；理想状态下由 AI 基于用户过程习惯自动归纳，再由人确认 | 流程时间线、checklist 卡片、历史经验模块 |

## 信息内容对象类型&语义

`信息内容与层级` 涵盖所有会进入最终版面的内容对象，按类型与信息角色分组如下：

| 分组维度 | 类别 | 包含对象 |
| --- | --- | --- |
| **对象类型** | 内容对象 | 文字 / 图片 / 图表 / 数据-数字 / 表格 |
| | 识别对象 | 图标 / Logo-标识 / 标签-徽标 / 二维码 |
| | 动作对象 | 按钮-CTA |
| | 结构对象 | 容器-卡片 |
| **对象语义** | 主信息层 | 主标题 / 副标题 / 核心结论 / 关键信息 |
| | 支撑信息层 | 支持证据 / 解释说明 / 对比项 / 案例内容 / 来源标注 |
| | 动作与品牌层 | 品牌标识 / 行动入口 / 辅助装饰 |

最小字段：`对象类型 | 对象语义 | 重要程度 | 改动权限`

<br />

## 跨层映射规则

跨层映射规则不作为主表中的独立节点，而作为连接 `信息内容与层级`、`结构与模块关系`、`视觉风格与参考锚点` 三层的隐性逻辑存在：信息内容的形态会影响结构选择，结构关系又会影响视觉表达方式。这种对应判断主要体现在设计原则节点中，而不是单独展开的规则列表。

<br />

## 底层规则与引用层

底层规则不作为元设计空间的主节点，原因有三：

- `与任务实例强耦合`：通常只对当前这张图、这个版式、这次交付成立，跨任务复用的稳定性低
- `抽象成本高于收益`：全部前置定义反而增加负担，并限制 AI 的执行灵活性
- `承载方式多样`：底层规则有两种存在形态——预存在 Assets 中可复用调用的，以及设计师在迭代过程中即时给出的修改指令；两者都不适合作为前置配置的主节点

此外，底层规则中有相当一部分属于**隐性知识**，本质上难以被规范化。P7 从工程实践角度给出了非常直接的描述：

> 哪怕组件都调对了，它还是会有一些像间距、排列的布局的结构、很细的小点，呼吸感、留白，这些地方对齐没有，没有贴紧。这些地方都是人要求的，哪怕你调用组件也只能解决一部分的问题。你要扣的那一些点，它不是能够通过 rules 或规则来描述清楚的，也不是模型读了就能理解以后能准确执行的。有一些非常隐性的知识，它难以规范化。

这意味着，即便投入大量成本去显性化底层规则，也无法兜住最后那部分只能靠人眼判断的细节——这正是 RQ2 聚焦中高层规则、而不是穷尽所有执行细节的核心理由之一。

底层规则的两种承载方式：

| 类别 | 性质 | 典型内容 | 示例 |
| --- | --- | --- | --- |
| 参考资源 | 预存型，被动引用 | 参考图 / 旧模板 / 风格库 / 设计规范 | 引用旧版海报模板对齐风格方向 |
| 可执行规则包 | 预存型，主动运行 | skill / 操作套路 / 版式配方 / 预设方案 / 检查清单 | 调用一套已有的排版方案直接展开 |
| 迭代修改指令 | 过程型，当次有效 | 设计师在执行阶段给出的即时细节要求 | "这个色块换成深蓝"、"标题字号再大一级"、"把这个模块移到左侧" |

<br />

## 中期原型范围

中期 demo 优先实现前六项，流程资产与经验沉淀放到后期。`Assets / 引用层` 可作为系统入口保留，但不计入 RQ2 主节点。

| 优先级 | 维度 | 说明 |
| --- | --- | --- |
| `高` | 任务目标与场景 | 需显式前置平台规格 / 比例 / 渠道，直接影响后续内容、结构与视觉决策 |
| `高` | 设计原则 | 补上跨任务、跨层级的高层判断标准 |
| `高` | 信息内容与层级 | 核心元设计层，直接决定传播重点 |
| `高` | 结构与模块关系 | 内容组织骨架，影响信息清晰度与修改成本 |
| `高` | 视觉风格与参考锚点 | 风格对齐依据，降低视觉方向偏差 |
| `中` | 人机分工与控制边界 | 体现系统区别于普通 prompt 面板的核心特征 |
| `后期` | 流程资产与经验沉淀 | 做沉淀与复用，不计入中期范围 |
