# RQ2

> 当前用途：作为 `RQ2` 的高频版工作文档，直接服务原型信息架构、字段设计与中期范围取舍。

## 研究问题

**RQ2：在信息设计场景下，具体任务之外，设计师可以制定怎样的元设计空间？**

## 一句话判断

`RQ2` 不研究所有细节，而聚焦那些 `高影响、高歧义、高复用` 的中高层元设计空间内容。

## 规则分层判断

如果要把 `RQ2` 中讨论的元设计空间进一步说明清楚，一个更直接的说法是：

- `高层规则`
定义任务前提、目标方向与总体判断标准，回答“这次设计为什么做、为谁做、什么算成立”
- `中层规则`
定义信息如何被组织、模块如何被编排、风格如何被锚定，回答“这次设计应如何被组织与展开”
- `底层规则`
定义局部执行方式与具体实现细节，回答“哪些元素怎么画、怎么排、怎么具体落出来”

对于本研究来说，`RQ2` 的重点主要放在 `高层规则 + 中层规则`，而不是穷尽所有底层细节。底层规则并非不重要，而是更接近具体任务实例、局部样式和执行展开，通常更适合作为：

- 被调用的执行约束
- 可复用的局部 recipe
- 模板、preset、skill、checklist 等外部化载体

换句话说，`RQ2` 更关心的是：`哪些规则值得被前置定义为元设计空间`；而不是：`所有具体执行细节都应被完整写死。`

## 主表与规则分层的对应关系

为避免读者在主表中自行推断三层关系，这里先明确映射一次：

| 规则层级 | 在本研究中的主要作用 | 对应的 RQ2 节点 |
| --- | --- | --- |
| `高层规则` | 定义任务前提、协作目标、评判标准与总体控制边界 | `任务目标与场景`、`设计原则`、`人机分工与控制边界` |
| `中层规则` | 定义内容组织、结构编排、风格锚点与跨层映射关系 | `信息内容与层级`、`结构与模块关系`、`视觉风格与参考锚点`、`跨层映射规则` |
| `底层规则` | 定义局部视觉执行、版式细节、素材调用与重复性实现方式 | 不作为 `RQ2` 的主节点展开，更多寄存在 `Assets / 引用层`、模板、preset、layout recipe、局部执行约束中 |

这也意味着，`RQ2` 当前文档里的主表并不是按“抽象概念并列罗列”来组织的，而是在事实上已经覆盖了 `高层 -> 中层 -> 底层载体` 的关系；只是此前这层分层逻辑还没有被显式说出来。

## 元设计空间主表

| 元设计空间维度 | 核心内涵 | 可结构化内容 | 典型形成方式 | 原型转译方向 | 阶段优先级 |
| --- | --- | --- | --- | --- | --- |
| 任务目标与场景 | 界定任务前提、使用语境与交付边界 | 受众、场景、渠道、目标、规格 | 由设计师直接设定，或通过与 AI 的前期对话共同澄清 | 任务卡片、场景面板、任务摘要条 | `高` |
| 设计原则 | 沉淀跨任务复用的高层判断标准，兼顾生成原则与评判标准 | 原则内容（生成&评判）、适用范围、关键映射判断 | 由设计师主动输入，并可在任务过程中持续抽象、补充与修正；AI 可辅助归纳但不替代人的原则判断 | 原则卡片、原则列表、可新增原则入口 | `高` |
| 信息内容与层级 | 明确任务中包含哪些信息，以及这些信息的主次关系 | 内容项、语义类型、重要程度、改动权限 | 由生成结果中识别与抽取，再由设计师进行审查、修正与补充 | 信息卡片列表、层级树、内容清单 | `高` |
| 结构与模块关系 | 明确内容如何被分区、排序、组织与打包 | 模块划分、模块顺序、模块主次、模块内对象组织 | 可由设计师直接组织，也可基于生成结果进行结构识别与后续修正 | 结构画布、模块编排板、结构关系图 | `高` |
| 视觉风格与参考锚点 | 提供风格方向及视觉对齐依据 | 风格关键词、参考图、主视觉方向、风格元素 | 可在生成前以前置风格锚点限定，也可在初步结果基础上追加风格约束 | 风格卡片、参考图板、轻量风格摘要 | `高` |
| 人机分工与控制边界 | 明确不同层上的主导权、介入程度与不可改动部分 | 人机参与度、锁定项 | 跟随信息、结构、视觉等层同步调节；AI 可先给出默认建议，再由设计师修正 | 分层控制栏、参与度调节、锁定标签 | `中` |
| 流程资产与经验沉淀 | 沉淀可复用的推进方式与关键检查经验 | 流程名称、适用场景、步骤列表、检查重点 | 原型阶段以人工编排为主，AI 可辅助总结；理想状态下由 AI 基于用户过程习惯自动归纳，再由人确认 | 流程时间线、checklist 卡片、历史经验模块 | `后期` |

## 主表之外只补四类信息

主表已经承担了核心分类。主表之后不需要再堆很多平行表格，保留四类补充信息就够了：

1. `信息对象最小定义`
2. `跨层映射规则`
3. `Assets / 引用层`
4. `中期原型范围`

这样可以让后面的内容都服务于主表，而不是跟主表再做一轮重复展开。

## 信息对象最小定义

`信息内容与层级` 这里说的不是只有文字，而是所有会进入最终版面的内容对象。

对象分组：

- `内容对象`
`文字` / `图片` / `图表` / `数据-数字` / `表格`

- `识别对象`
`图标` / `Logo-标识` / `标签-徽标` / `二维码`

- `动作对象`
`按钮-CTA`

- `结构对象`
`容器-卡片`

角色分组：

- `主信息层`
`主标题` / `副标题` / `核心结论` / `关键信息`

- `支撑信息层`
`支持证据` / `解释说明` / `对比项` / `案例内容` / `来源标注`

- `动作与品牌层`
`品牌标识` / `行动入口` / `辅助装饰`

最小字段：

`内容项 | 语义类型 | 重要程度 | 改动权限`

## 跨层映射规则

这部分不作为主表中的独立节点，而作为连接规则存在，主要连接：

- `信息内容与层级`
- `结构与模块关系`
- `视觉风格与参考锚点`

它真正回答的是三件事：

- 什么内容形态适合什么结构组织
- 什么结构关系适合什么视觉表达
- 哪些对象组合一出现，就应优先匹配哪类模块、版式或呈现方式

典型例子：

- 有真实量化数据时，更适合映射到 `KPI / 图表 / 排名` 一类结构，而不是纯宣言式表达
- 以截图或照片作为证据时，更适合映射到 `证据墙 / 图文并置 / 对照` 一类结构
- 以核心结论为主的信息层级，更适合映射到 `高强调标题 + 收束型模块` 的视觉策略

主要挂载位置：

- `设计原则`
- `结构与模块关系`
- `视觉风格与参考锚点`

## Assets / 引用层

这部分不再视为 `RQ2` 的元设计空间节点，而更适合作为系统中的一层辅助能力。这里建议只区分两类：

- `参考资源`
更偏被动引用，例如：

- `参考图`
- `旧模板`
- `风格库`
- `设计规范`

- `可执行的元设计包`
更偏主动运行，例如：

- `skill`
- `template`
- `preset`
- `layout recipe`
- `checklist`

两者的区别是：后者不只提供参考，还已经把一部分设计经验封装成了可直接调用的机制，例如：

- 可选模板
- 版式配方
- 平台适配规则
- 默认约束
- 检查要点

因此，`Assets / 引用层` 更适合理解为：

- 输入入口
- 引用层
- 辅助表达机制
- 外部化的规则载体

而不是原设计空间主节点本身。

## 中期原型最小范围

中期 demo 最值得先做的六项：

1. `任务目标与场景`
2. `设计原则`
3. `信息内容与层级`
4. `结构与模块关系`
5. `视觉风格与参考锚点`
6. `人机分工与修改权限`

取舍原则只保留四条最关键判断：

- 前五项负责把 `RQ2 = 元设计空间` 讲清楚，其中 `设计原则` 负责补上跨任务、跨层级的高层判断
- 第六项负责体现这不是普通 prompt 面板，而是可在各层细调人机介入比例的系统
- `任务目标与场景` 中应显式前置 `平台规格 / 比例 / 渠道机制`，因为它们会直接影响后续内容、结构与视觉决策
- `Assets / 引用层` 可作为系统入口保留，但不计入 `RQ2` 主节点
- 第七项放到后期做沉淀与复用

## 页面级启发

如果把它翻成一个中期页面，最自然的结构仍然是六区：

1. `受众 / 场景 / 渠道 / 目标区`
2. `设计原则区`
3. `内容与层级区`
4. `结构与模块区`
5. `风格与参考区`
6. `协作控制栏`

## 最终收束

这份文档真正想支持的不是“用户如何写一句更好的 prompt”，而是：

`如何把元设计空间中的关键节点做成可见、可配、可加载、可复用的原型对象。`

同时，这个版本还明确补上三件此前容易散落在表格里的内容：

- `跨层映射规则`
- `平台规格带来的前置交付边界`
- `以 skill / template / preset / checklist 形式存在的可执行元设计包`
