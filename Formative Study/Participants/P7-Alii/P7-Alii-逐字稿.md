阿礼 Alii（00:02:22）
我本职是交互设计，然后我在的业务是做 AI coding 产品线的，说白了我在做 Codex 的内部产品，就平行的竞品。我从24年开始就用 Cursor，到现在26年我已经常态是用 Cursor 加 Codex 的组合，直接在工作里面做产品需求，直接从设计稿到开发，跟研发一起提交代码 PR 要合并上线这样的一个路径了。

阿礼 Alii（00:03:36）
至少我们这一边，比较年轻的，工作一两年两三年的，已经能够很快的直接把设计交付变成代码交付。对 AI coding 上手的程度要学一点点代码知识，就变成了从交付设计稿变成直接交付代码，目前是这样的情况。

主持人（00:03:56）
就你交付的代码还会有前端去再做审查吗？

阿礼 Alii（00:04:02）
有需要前端审查，差异。例如说我们去优化改图标、优化一下素材、优化一下文案间距、圆角、hover 状态，review 很快，研发可能看一眼就 ok 了。还有一种，我要做体验优化，比如说线上有一个地方，任务的侧栏有很多状态的图标，包括它响应的位置是不对的，这时候是样式和交互一体的优化，改动不可避免会涉及到逻辑。这个时候只能设计师来改，但你得先能争取到研发的支持，自己具备一定的通过代码协同的能力，知道怎么提交 git，怎么自己先本地自测验证代码的可用性和质量。

主持人（00:05:05）
能感受到基本上已经完全具备了设计加工程的能力了。

阿礼 Alii（00:05:12）
Design engineer 这一块。

阿礼 Alii（00:05:17）
他可能对设计师本身的要求会有点高。我能跑通的前提是我的作图习惯真的非常规整。我的设计稿是跟前端的盒子模型一样，我有全组件加全 auto layout。我自己做图的时候，我的脑子里对界面交互状态的构思已经想了一半了，代码只是帮我把这一块——我已经想的非常清楚、交互标注的非常清晰的东西——给转化成代码而已。

阿礼 Alii（00:05:51）
如果是那些比较视觉出身的设计师，他本身画图的情况可能比较偏图层逻辑，图层比较多，打组很多蒙层，他传递的时候——无论是你直接 Figma D2C 让 AI 帮你读设计稿，还是截图去图它的还原效率——在你的结构没有想清楚，或者不能足够清晰阐述清楚的情况下，会导致不能像有研发背景的设计师一样，精细地掌控界面的表现能力。

主持人（00:06:53）
了解，那我还好奇，前期类似于探索、AB 稿这种过程你们还存在吗？

阿礼 Alii（00:07:05）
这里我们还在讨论。从生成载体上来说，现在已经分割成两个流派了——从纯设计 first（Figma 那类，DSL 节点，以页面效果为优先），到完全代码 first（直接生码，受 DOM 树和 CSS 约束）。中间有一个过渡地带，有非常多产品在尝试。比如 Claude Design 偏代码那一点，生成的是 HTML；Lovable、v0 那类生成的是 React 代码，改起来非常费劲。

阿礼 Alii（00:10:01）
我们内部在推说怎么让一些比较轻度的设计师、包括产品经理运营这一类泛产业角色也参与到产品设计的 AI 辅助中来，变成了分歧点。

阿礼 Alii（00:11:48）
在我们真实工作中，方案发散的场景没有那么多，因为整个需求的大部分比例都是在既有的页面里去迭代。从0到1的新模块和新页面，每个季度也就那么几次。做完之后，后面都是在既有的页面上做迭代，这时候要考虑的就不是很大方向的发散，而是要知道既有产品的上下文——无论是视觉规范、交互逻辑、整个页面的路径、整个信息架构——然后在这些既有信息串联的约束之下，做很细的一层交互发散。

阿礼 Alii（00:13:14）
AI 对既有信息、既有上下文的耦合度不高，它比较独立，是从0到1的，AI 能够很好的提供一些帮助。但凡它涉及到深水区，涉及到一到N的复杂模块的维护和串联、多种上下文的判断，这个时候无论你输入多大段的 prompt、截图、rules，都是很累很费劲的。费劲程度就变成了——不如我还是回归传统流程，自己想，跟同事真人沟通打磨。

主持人（00:13:57）
你平时还会给 AI 一些类似于设计或者业务的 context 吗？

阿礼 Alii（00:15:33）
例如一个完整的需求，我最近在优化 agent team 的需求——需要在某个页面上做一些新的模块，工程师已经将逻辑代码做好了，然后我需要在他的逻辑代码上做 UI。这时候我是直接拿到了代码仓库，让 AI 读代码，调研它已经在代码仓库里实现好的东西，然后告诉我它的 UI 展示是怎么实现的，定位到代码片段在哪。接下来我就把设计稿设计出来，再告诉 AI 怎么怎么样地实现。

阿礼 Alii（00:17:53）
我的代码语言偏向是这样的：在调 UI 细节的时候，我会用 Tailwind 的语法，非常细。像更换图标，我会直接让 AI 替换 SVG，指定在 Figma 里贴的那个 SVG。有一些用词，为了方便对齐模型去理解，会用代码里面直接的名字，包括变体、这些映射，图标的状态什么的。这就有一些偏工程化设计思维里面说的那种：图层等于一个 div，组件等于代码的组件，设计稿的变体等于代码里面的 property。

主持人（00:19:37）
假如说要维护更新其中一个组件，对你来说困难吗？效率高吗？

阿礼 Alii（00:20:00）
如果是业务里面的，我通常就不会让 AI 来 vibe 了。如果只是去应付一个报告，用 AI 随便 vibe 一下也可以，我不关注细节。如果这个东西是我的毕业答辩的讲稿，我肯定需要字斟句酌地把每一个细节都考虑到位。如果设计稿只是凑合用的、临时的，用 AI 随便 vibe，我不关注细节。如果是工作里面的稿件，AI 还不具备对细节量的把控，这时候还是回归到我在设计稿里面把每一个细节的动态都调好，再把确定好的参数转述给 AI。

主持人（00:44:03）
你平常用到场景还比较多，比如小红书的图文，你会尝试沉淀一套自己的设计规范风格吗？

阿礼 Alii（00:44:35）
我的沉淀是设计系统这一套设计 token。例如这里有一套从代码到自己搞都对齐的 token，包括一些组件规范。一套色，亮暗两套，然后放到代码里。每一个颜色的亮暗都是对应的。如果这里就只说工作场景，你会见到界面上有深深浅浅的灰、深深浅浅的黑、深深浅浅的白。

主持人（00:50:19）
你是有非常有元设计思维的一位设计师。什么是元设计？简单来说就是在做具体的设计之前，会先去定一套规则、规范和流程，再上手的时候运用这套规则。您对于在设计之前专门维护这么一套规范规则，有怎样的一些想法和洞见？

阿礼 Alii（00:51:22）
我觉得这东西是当你的某一个业务或作品或 project 它大到一定程度必然需要的一个东西。不然你那个东西就会跟 AI 抽奖一样，每次都不稳定，不一致。维护这一套，跟你在 coding 领域工程师可能会去维护一套开发规定——写在代码仓库的 markdown 里，跟 AI 说哪些模块不能改、哪些模块能改——是一样的用途。当它大到一定程度之后，这一套规范是替代你每轮坐在电脑前去给他 feedback 的，一个异步的、离线的、常驻的约束规则。

阿礼 Alii（00:52:26）
如果是复杂的需求，我还需要回到设计稿，先画出来，然后再给 AI 灌输我的 prompt。这也是一种——当需求要考虑的复杂度上了一层，他就不能简单地通过纯对话去解决，那一层人和 AI 之外的第三份产物，那份常驻的像小本本一样的产物，也是一种可以固化的上下文。

主持人（00:53:25）
假如说你是一个在小红书经常分享 AI 知识的博主，你需要维护一份关于小红书封面图文的这样一个 AI 设计规范，你会尝试维护哪些内容？

阿礼 Alii（00:53:55）
我可能会在小红书的 skill 里面，我在 GitHub 扒到了一个，skill 里面已经有一套 HTML 的模板了。然后调过一轮的时候，我已经有意识地在说，你把所有的 header 和 footer 都变成一个组件，每个页面都用同一个。包括配色，我是能看到 HTML 的。我可能会把它稍微改一改，变成更加有个性化的、我个人特色的一些色值，让它的风格跟开源的样式有一些区别。

阿礼 Alii（00:54:47）
我还可能会让他截取文案，在转化的时候有一些规则，例如我希望他更加使用子弹清单题，更加简练，更加重点突出。还有如果是生图的话，因为我知道他几斤几两，我可能会说你不行，你就申请个占位符——与其他自己生，不如做给我插入的友好的界面。

主持人（00:57:00）
如果有这种需要一直让你去维护这一套规则的情况存在，还会有怎样的隐患？

阿礼 Alii（00:57:13）
维护我是可以接受的。小红书封面这种相对比工作场景来说已经算是很轻量了，调好之后差不多就没什么大的改动了，可能会一直这样用。工作场景，维护会非常重，要捋很多很重的样式、素材、图标、组件，要写很长的那些被调用组件的 rules，很重。然后我们就这一块，反而我都觉得很头疼，我甚至是在另外一派的。我们同事花了几个月做很重的规范，我这边我是直接去写写代码的人，我觉得与其把规则写在 markdown 里，不如写在脑子里，反正都是最终都要我校验，那不如我今天就自己校验得了。

阿礼 Alii（00:58:47）
哪怕组件都调对了，它还是会有一些像间距、排列的布局的结构、很细的小点，呼吸感、留白，这些地方对齐没有，没有贴紧。这些地方都是人要求的，哪怕你调用组件也只能解决一部分的问题。你要扣的那一些点，它不是能够通过 rules 或规则来描述清楚的，也不是模型读了就能理解以后能准确执行的。有一些非常隐性的知识，它难以规范化。

阿礼 Alii（00:59:34）
这就是为什么我觉得虽然我们同事在做规范，他们做的那么辛苦，调效果也就那样，最终还是要人来修。那不如自己去，作为人类最后把关的人去修。因为反正每次哪怕你分的再细，提交的时候，需要有个人在那里。最终还是要有个人在那里的。有些时候设计师自己都调完了，研发都不告诉你就上线了，得接受。有些不完美的存在，甚至可能用户都不 care。

阿礼 Alii（01:00:30）
在有限的注意力资源之下，尽可能把控必要的细节是不出错的。其他地方，在能把控的情况下、时间允许的情况下尽快把控；把控不了，就只能允许他先上线了。

阿礼 Alii（01:04:22）
我之前在公司有一个分享，里面讲到了 harness 的策略——怎么去通过设置层层的约束，尽可能让 AI 生成的效果符合我们的意愿。说的是从代码侧和设计侧，为了让它生成代码更加可用、更加符合产品规范，有哪些约束。从上到下分别是完全可以精细地让代码或 AI 来执行，最下面是非常模糊、非常难以具象化，以至于说不清楚，需要人工 review。

阿礼 Alii（01:05:35）
如果是代码的话，它可能会有单元测试，然后字符级别的检测，非常清晰，可以写在代码里让它自己跑。再抽象一些，像代码架构的规则，情况很多，可以写成一个 rules 让 AI 语义化理解和执行——约束从100%硬变成了稍微软了一点，有可能漏网。再抽象一点，一些更宏观的架构，你的描述可能会更加难以确保 AI 每次都理解对。这里就有一个从完全具象到不具象的过程。

阿礼 Alii（01:06:36）
第二个是每一个约束措施有一个显性化成本的考虑——怎么把本来是专业人士的知识，给显性化成清晰的 agent 或代码可以执行的规则。设定出来之后，每次更新要去维护。非常清晰的就很低——token 是非常清晰的，然后是比较容易显性化出来。你再宏观一点，可能到组件库，组件库可能会随着业务新模块而变化。再宏观一点，像交互模式库——一个订单采购的路径是怎么样的，它大概也是有些规则的，有些约定俗成的模式库。再泛一点，可能就是很细的编剧、对齐、布局，视觉这层级的留白设置——这个留白不太舒服，我很难把它写成一个 rules 让他知道，然后来遵循它。已经进入到了难执行、甚至只能意会说不清规则的地步了。

阿礼 Alii（01:08:02）
如果它半衰期越短，显性化成本越高，那是不是说明它可能就不适合让 AI 来执行，而适合人在 loop 里面去记录一下？这个时候就非常适合人工 review 了。

阿礼 Alii（01:08:51）
Token 那层是100%可以遵循的，但到了交互那些东西太抽象了，到了页面级的情感化，就确实得人来看，影响的约束程度。还有另外一个点——模型本身的重力，早一点的模型，4.6效果都没那么好，4.7才开始说，可能有一些说不清的规则，它本身自己也会注意了。Claude 或 OpenAI 他们家的模型，其实是在随着迭代将社区里比较好的那种写在 rules 里的规范慢慢给内置到模型训练里面。就会变成原本你可能需要外置的一个 skill 来告诉他怎么样，到后面你不需要 skill，他也能做好。

阿礼 Alii（01:10:13）
最惨最后那个兜底之后，可能10%非常重要的细节，在你在意的产品上、在最关键页面上最关键的物料上，还是需要人去注意。如果大家都在用 AI 了，你是团队里面唯一的那一两个用来把关设计质量的人，你只能去确保最关键的那些给用户看到的物料是没有出问题的。
