# 核心价值讨论

## 课题定位

本课题关注的不是终端设计师如何直接使用生成式 AI，
而是面向信息设计场景，探索机制设计者如何预先配置设计意图表达、生成规则、过程约束与人机分工，
从而支持后续更可控的人机协作生成。

## 核心价值

本课题的核心价值，不是做一个“大而全”的 AI 设计系统，
也不是单纯优化 prompt，
而是把生成式信息设计中的协作机制本身，提升为一个可以被预先设计和配置的对象。

## 关键角色

- 机制设计者 / 元设计师：负责定义系统中哪些内容可以被表达、哪些规则可以被修改、哪些步骤由人或 AI 完成
- 终端设计师：在这套预先设计好的协作机制中，与 AI 一起完成具体的信息设计任务

## Gap

现有生成式设计工具大多面向“使用时交互”，
强调用户如何即时输入需求并获得结果，
但较少支持由机制设计者预先配置：

- 设计意图的表达维度
- 生成规则与约束
- 过程规则
- 人机分工方式

也就是说，现有工作更多在优化“如何用”，
较少关注“协作机制本身如何被预先设计”。

## 可能的贡献

1. 将生成式信息设计中的协作机制本身作为设计对象
2. 探索哪些设计意图维度适合被显式化和机制化
3. 探索生成规则、过程约束与人机分工如何被预先配置

## 对终端设计师的价值

终端设计师不是每次都从零开始和 AI 协作，
而是在一套已经被设计过的机制里工作。

这样做的好处是：

- 不用每次重新摸索怎么和 AI 配合
- 生成结果更稳定，不容易跑偏
- 减少反复改 prompt 的试错成本
- 可以复用资深设计师沉淀下来的经验

## Q&A

### 和直接写 skill 的区别是什么？

skill 当然也可以定义规则，但它更像是一套已经写好的执行逻辑。
一旦规则、流程或者人机分工需要调整，往往就要回到 prompt 或代码里重新修改。

相比之下，这个平台更像一个持续配置和调整协作机制的环境。
它的价值不只是“把规则写出来”，而是让这套规则更容易被修改、复用和共享，
也更容易被团队中的其他人理解和使用。

所以，相比直接写 skill，这个平台更适合支持团队内部长期维护的一套生成式信息设计协作机制。

## 一句话版本

本研究不是研究终端设计师如何更好地下 prompt，
而是研究机制设计者如何预先设计一套生成式信息设计的人机协作机制，
使后续设计任务能够在其中被更可控地生成与调整。
